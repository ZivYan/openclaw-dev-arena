# 02 - Phase 0：需求锁定

## 目标

在写代码前，把任务边界锁死。

## 默认执行角色

- 角色：`rd-lead`
- 推荐模型：Claude Opus

## 必须产出

- **目标**：真正要解决的问题
- **非目标**：这次明确不做什么
- **影响范围**：允许修改哪些文件/模块
- **约束条件**：兼容性、性能、时限、依赖限制
- **验收标准**：什么结果算完成

## 必问问题

1. 这次要解决的根问题是什么？
2. 哪些文件或模块允许改？
3. 哪些内容明确不能动？
4. 用户最看重的成功标准是什么？
5. 这是临时修复还是长期方案？

## 通过标准

- 需求能用 3-5 句话复述
- 验收标准可以直接验证
- 没有关键歧义

## 输出模板

```markdown
## 任务定义
- 目标：
- 非目标：
- 影响范围：
- 约束：
- 验收标准：
```
