# 01 - Skill 架构与角色路由

## 核心模型

这个 Skill 的核心不是让一个模型做完所有事，而是**对外单入口、对内多职责角色**：

- **`rd-lead`**：需求锁定、方案对抗、任务拆分、对抗评审
- **`rd-coder`**：编码实现、自测执行、问题修复
- **`rd-ship`**：验证复核、提交交付、报告整理

## 推荐模型映射

- `rd-lead` → Claude Opus
- `rd-coder` → Codex
- `rd-ship` → GPT-5.4

## 单入口原则

- 对用户：默认只暴露 `rd-lead`
- 对系统：按阶段把工作转发给内部角色
- 对交付：所有阶段结论再回到 `rd-lead` 做裁决

## 总流程

```text
Phase 0 需求锁定
Phase 1 方案对抗
Phase 2 任务拆分
Phase 3 原子开发
Phase 4 自测验证
Phase 5 对抗评审
Phase 6 提交交付
```

## 阶段路由

| Phase | 默认角色 | 说明 |
|------|---------|------|
| 0 需求锁定 | `rd-lead` | 锁目标、非目标、约束、验收标准 |
| 1 方案对抗 | `rd-lead` | 双方案对比、反驳、裁决 |
| 2 任务拆分 | `rd-lead` | 拆成原子任务，定义边界 |
| 3 原子开发 | `rd-coder` | 改代码、补测试、处理实现细节 |
| 4 自测验证 | `rd-coder` 主执行，`rd-ship` 复核 | 跑命令、汇总结果、检查遗漏 |
| 5 对抗评审 | `rd-lead` | 从 P0-P3 角度否定实现 |
| 6 提交交付 | `rd-ship` 主输出 | 整理 commit / PR / 交付说明 |

## 运行时实现

推荐实现：

```text
用户
  ↓
rd-lead (Claude Opus)
  ├─ sessions_send → rd-coder (Codex)
  ├─ sessions_send → rd-ship (GPT-5.4)
  └─ 汇总结果后给用户结论
```

如果运行时支持 `sessions_spawn`，优先用它拉起 `rd-coder`、`rd-ship`；否则给它们各保留一个内部专用 session。

## 质量优先级

1. 安全性
2. 正确性
3. 可测试性
4. 一致性
5. 可维护性
6. 开发效率

## 适用范围

- 新功能开发
- Bug 修复
- 测试补充
- 文档同步
- 代码评审
- 提交前收口

## 默认原则

- 先锁需求，再给方案
- 先定义验收，再写实现
- 优先最小改动
- 不做无关重构
- 不忽略错误与验证
- 结论必须能追溯到命令、文件或测试结果
