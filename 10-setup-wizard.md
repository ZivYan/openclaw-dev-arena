# 10 - 端到端执行清单

## 角色准备

- [ ] 配置 `rd-lead`，模型建议用 Claude Opus
- [ ] 配置 `rd-coder`，模型建议用 Codex
- [ ] 配置 `rd-ship`，模型建议用 GPT-5.4
- [ ] 仅把 `rd-lead` 暴露给用户
- [ ] 给 `rd-coder`、`rd-ship` 准备内部 session

## Phase 0：需求锁定

- [ ] 由 `rd-lead` 执行
- [ ] 明确目标
- [ ] 明确非目标
- [ ] 明确影响范围
- [ ] 明确验收标准
- [ ] 需要时向用户确认

## Phase 1：方案对抗

- [ ] 由 `rd-lead` 执行
- [ ] 给出方案 A：最小改动
- [ ] 给出方案 B：结构优化
- [ ] 写清两边风险
- [ ] 说明最终选择理由
- [ ] 需要时向用户确认

## Phase 2：任务拆分

- [ ] 由 `rd-lead` 执行
- [ ] 拆成原子任务
- [ ] 每个任务可独立验证
- [ ] 超过 5 个文件时创建 `IMPLEMENTATION_PLAN.md`

## Phase 3：原子开发

- [ ] 由 `rd-coder` 执行
- [ ] 先找相似实现
- [ ] 先改核心入口
- [ ] 再补测试或验证
- [ ] 再同步文档

## Phase 4：自测验证

- [ ] `rd-coder` 跑命令
- [ ] `rd-ship` 复核结果
- [ ] 跑最小范围验证
- [ ] 再跑更大范围验证
- [ ] 记录命令与结果

## Phase 5：对抗评审

- [ ] 由 `rd-lead` 执行
- [ ] 检查 P0 安全问题
- [ ] 检查 P1 稳定性问题
- [ ] 检查 P2 正确性问题
- [ ] 检查 P3 可维护性问题

## Phase 6：提交交付

- [ ] `rd-ship` 负责输出交付材料
- [ ] `rd-coder` 提供命令与修改事实
- [ ] 说明修改内容
- [ ] 说明验证结果
- [ ] 说明风险与下一步
- [ ] 按需提交 / push / PR
